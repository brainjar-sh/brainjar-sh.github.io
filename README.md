# brainjar.sh

Source for [brainjar.sh](https://brainjar.sh) — the documentation site for [brainjar](https://github.com/brainjar-sh/brainjar).

Built with [Astro](https://astro.build) + [Starlight](https://starlight.astro.build), deployed to GitHub Pages on every push to `main`.

## Layout

```
src/content/docs/
├── index.mdx                 # landing page
├── getting-started.md
├── concepts/                 # soul, persona, rules, procedure, brain, …
├── guides/                   # mcp, hooks, packs, recipes, …
└── reference/cli.mdx         # autogenerated — gitignored
public/
├── llms.txt                  # agent-readable doc index
├── versions.json             # written at build time by fetch-versions.sh
└── versions.js               # version-bar widget
scripts/
├── fetch-cli-docs.sh         # download docs.tar.gz for the latest brainjar release
├── fetch-versions.sh         # write public/versions.json from latest release tag
└── copy-raw-md.sh            # mirror src/content/docs/**/*.md into dist/ at <page>.md
```

## Build pipeline

Every build does three things in addition to running Astro:

1. **`fetch-versions.sh`** — fetches the latest brainjar release tag from `https://get.brainjar.sh/brainjar/latest` and writes it into `public/versions.json`. The version bar in the hero reads this client-side.
2. **`fetch-cli-docs.sh`** — pulls `docs.tar.gz` for the same tag and consolidates every cobra-emitted command page into a single Starlight page at `src/content/docs/reference/cli.mdx`. The site never reads a local `brainjar` binary; the source of truth is the release artifact. To preview docs against an unreleased CLI change, cut a prerelease tag on the brainjar repo and set `BRAINJAR_DOCS_TAG` before running `npm run build`.
3. **`copy-raw-md.sh`** — after Astro builds, copies the raw markdown sources into `dist/` so URLs like `brainjar.sh/getting-started.md` serve the raw content. Combined with `public/llms.txt`, this makes the whole site agent-readable.

## Local development

```bash
npm install
npm run dev      # http://localhost:4321
```

`npm run dev` and `npm run build` both run the fetch scripts first, so a local checkout always renders against the *currently published* CLI reference.

| Command | Action |
| :--- | :--- |
| `npm run dev` | Starlight dev server at `localhost:4321` |
| `npm run build` | Production build into `dist/` |
| `npm run preview` | Serve the built `dist/` locally |

## Deploy

`.github/workflows/deploy.yml` runs `npm ci && npm run build` on every push to `main`, then publishes `dist/` via GitHub Pages. The `CNAME` in `public/` points the deployment at `brainjar.sh`.

## Updating docs for a new brainjar release

The CLI reference and version bar update themselves on the next site build — no doc-side action required for the autogenerated parts. For new features that need hand-written prose (concept pages, guides, examples), edit the relevant page under `src/content/docs/` and add a sidebar entry in `astro.config.mjs` if you're introducing a new top-level page.
